# 测试用例 Markdown 格式规范 (v2.0)

本文档定义了由 `Testcase-Writer-XMind4MS` Skill 生成的测试用例 Markdown 文件必须遵循的完整格式规范。

## 一、 文档整体结构
一个完整的测试用例 `.md` 文件必须包含以下部分，顺序不可更改：

```markdown
# [系统或功能名称]测试用例

## [模块A名称]
- [子模块A1名称]
  - case: [用例标题A1.1]
    - 标签: [功能标签]
    - 前置条件: 1. [条件1]；2. [条件2]
    - 步骤描述:
        - 步骤: 1. [操作1]；2. [操作2]
          - 预期结果: 1. [结果1]；2. [结果2]
    - 用例等级: P1
- [子模块A2名称]
  - case: [用例标题A2.1]
    ...

## [模块B名称]
- [子模块B1名称]
  ...
```

## 二、 单个用例字段规范 (原子规则)
每个用例必须严格遵循以下结构，字段顺序不可更改：

```markdown
# [系统或功能名称]测试用例

## [模块路径]
- [子模块名称]
  - case: [用例名称]
    - 标签: [功能标签]
    - [前置条件: [条件描述]]  # 可选字段
    - 步骤描述:
        - 步骤: [操作步骤描述（使用1. 2. 格式）]
          - 预期结果: [结果描述（使用1.；2.；格式）]
    - 用例等级: [P1/P2/P3]
```

### 字段定义
1.  **文档标题** (必填)
    - 格式：`# <系统或功能名称>测试用例` (如：`# 操作手册管理测试用例`)
    - 要求：作为 XMind 根节点标题，必须来自用户需求中的系统、模块或功能名称。不要使用泛化标题，如“测试用例思维导图”，除非需求中完全没有可识别的业务名称。

2.  **模块路径** (必填)
    - 格式：`## 一级模块-二级模块` (如：`## 绩效管理-试用期考核`)
    - 要求：反映功能在系统架构中的位置。`##` 标题本身就是模块路径，标题下禁止再输出同名的 `- 模块路径` 列表项。

3.  **子模块名称** (必填)
    - 格式：`- <功能子模块>` (如：`- 权限验证`)
    - 要求：明确功能细分领域。

4.  **用例名称** (必填)
    - 格式：`case: <动词短语>` (如：`case: 权限验证-员工本人-默认展示与操作权限`)
    - 要求：必须包含“操作对象”和“验证点”，清晰无歧义。

5.  **标签** (必填)
    - 格式：`标签: <功能标签>` (如：`标签: 权限`)
    - 要求：用于分类和筛选。常见值：`功能测试`，`边界测试`，`异常测试`，`权限测试`，`UI测试`，`接口测试`。

6.  **前置条件** (可选)
    - 格式：`前置条件: <条件描述>`
    - 要求：用中文分号 `；` 分隔多个条件。 (例如：`前置条件: 1. 用户已登录；2. 拥有管理员权限`)
    - **省略规则**：若无前置条件，直接省略该行，不要输出空的 `- 前置条件:` 行。

7.  **步骤描述** (必填)
    - 格式：
      ```markdown
      步骤描述: 
          - 步骤: <操作步骤>
            - 预期结果: <验证点>
      ```
    - 要求：
        - **步骤**：使用 `1. [动作]；2. [动作]` 格式，句末用分号，最后一步可省略。
        - **预期结果**：使用 `1. [结果]；2. [结果]` 格式，句末用分号。
        - 每个用例只描述一个主要操作流程。

8.  **用例等级** (必填)
    - 格式：`用例等级: <P1/P2/P3>`
    - 等级定义：
        - **P1**：核心功能/高频场景/主干流程。必须优先覆盖。
        - **P2**：重要功能/边界场景/异常流程。重要但非阻塞。
        - **P3**：边缘场景/低频功能/体验优化。可后续覆盖。

## 三、 组织与编排规则
1.  **根节点标题**：文档第一行必须是一级标题 `# <系统或功能名称>测试用例`，用于控制 XMind 中心主题名称。
2.  **模块分组**：使用 Markdown 二级标题 `##` 对测试用例进行模块分组。标题名称为“模块路径”，例如 `## 绩效管理-试用期考核`。
3.  **用例列表**：在每个 `##` 标题下，必须直接从"子模块名称"列表项开始 (`- 子模块名称`)，并列放置该模块下的所有测试用例。禁止在 `##` 标题下重复输出同名模块路径，否则生成的 XMind 会出现第二层与第三层重复。
4.  **扁平模块处理**：若模块结构天然无子模块层级（即功能扁平，只有一个分组），则 `- case:` 直接置于 `##` 标题下，跳过 `- 子模块名称` 层级。此时 `case:` 上的 `-` 缩进为 2 空格（与同级 `- 子模块名称` 对齐）。
5.  **XMind默认展示**：转换脚本会将 XMind 根节点固定为右向逻辑图，并默认折叠 case 下的详情字段，以保持打开后的布局紧凑。
6.  **空行分隔**：
    - 文档开头不需要任何前言或说明文字，直接从 `# 标题` 行开始。
    - 不同 `##` 模块标题之间，用**一个空行**分隔。
    - 同一模块下的不同用例之间，**无需**空行。
7.  **排序**：
    - 模块按业务逻辑或需求中出现的顺序排列。
    - 同一模块下用例，按 **优先级 (P1>P2>P3)** 降序排列，同优先级按操作流程排列（正向流程 → 异常流程 → 边界流程）。

## 四、 质量检查清单
生成的每个用例在提交前应检查：
- [ ] 用例标题是否清晰明确了操作和验证点？
- [ ] 步骤是否具体、可执行、无歧义？
- [ ] 预期结果是否可观测、可验证？
- [ ] 是否覆盖了正常、异常、边界场景？
- [ ] 前置条件是否完整且必要？
- [ ] 标签和等级是否准确？
